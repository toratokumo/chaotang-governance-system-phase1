# 尚书省：朝堂系统技术落地方案

## 一、定位

此系统不是单纯展示数据的看板，而是用于后续各项目立项、推进、审批、监控、统计、归档的“朝堂操作系统”。

尚书省关注的是：
- 如何落地
- 如何分阶段实施
- 如何保证后续可扩展
- 如何让项目真正跑起来

---

## 二、建设原则

- 先能运转，再求完备
- 先可留痕，再谈智能
- 先定边界，再做扩展
- 先抓核心流程，再铺次级功能
- 权限、审计、统计必须从第一期就纳入

---

## 三、系统核心能力

### 1. 朝堂总览
首页应支持陛下一眼查看：
- 项目总数
- 进行中 / 阻塞 / 已完成 / 已延期
- 今日待决事项
- 高风险事项
- 最近告警
- 国库消耗概况
- 各部门负载概况

### 2. 项目治理
每个项目至少应具备：
- 名称 / 编号
- 发起时间
- 优先级
- 当前阶段
- 负责人
- 所属部门
- 截止日期
- 风险等级
- 附件与备注

### 3. 流程审批
基础流程建议统一为：
- 提案
- 草拟
- 审议
- 批准
- 执行
- 验收
- 归档

每一步都要记录：
- 处理人
- 处理时间
- 处理结果
- 批注
- 下一步流向

### 4. 国库统计
至少统计：
- 各项目 token / 预算消耗
- 各部门资源占用
- 日 / 周 / 月趋势
- 超额提醒

### 5. 告警监察
应具备：
- 延期告警
- 流程卡滞告警
- 超预算告警
- 长时间无人推进告警
- 权限异常告警
- 关键节点失败告警

### 6. 审计归档
必须记录：
- 立项
- 状态变更
- 审批结果
- 预算调整
- 权限变更
- 归档 / 反归档

---

## 四、技术架构建议

### 前端
采用 Web 控制台，主要页面包括：
- 朝堂总览
- 项目列表
- 项目详情
- 流程审批
- 国库统计
- 告警中心
- 审计日志
- 角色权限管理

### 后端
采用统一 API 服务，负责：
- 数据读写
- 权限校验
- 流程流转
- 告警触发
- 统计汇总
- 外部接口预留

### 数据库
建议使用关系型数据库，核心对象包括：
- users / roles
- projects
- tasks
- workflow_instances
- workflow_steps
- budgets / usage_records
- alerts
- audit_logs
- attachments

### 权限模型
建议使用 RBAC：
- 陛下
- 宰相
- 中书省
- 门下省
- 尚书省
- 六部
- 承办人

权限维度分为：
- 可查看
- 可编辑
- 可审批
- 可归档

---

## 五、实施阶段表

### 第一阶段：定制立规（1周）
交付：
- 模块清单
- 页面结构清单
- 数据字段初稿
- 角色权限矩阵
- 一期边界确认

### 第二阶段：原型与数据模型（1~2周）
交付：
- 页面原型图
- 数据库表结构草案
- API 草案
- 流程状态图

### 第三阶段：一期开发（2~4周）
交付：
- 登录与角色权限
- 朝堂总览页
- 项目列表页
- 项目详情页
- 基础流程流转
- 国库统计基础页
- 审计日志页

### 第四阶段：联调与试运行（1~2周）
交付：
- 测试问题清单
- 权限修订
- 告警规则初版
- 试运行反馈报告

### 第五阶段：二期增强（后续）
扩展：
- 更复杂审批流
- 甘特图 / 时间轴
- 自动催办
- 智能汇总
- 健康度评分
- 跨项目依赖分析

---

## 六、尚书省建议

最稳妥的推进顺序为：

**先定模块与权限 → 再做原型与数据模型 → 再开发一期最小可用版 → 最后逐步扩展。**

若一开始贪大求全，极易返工；若先做可立项、可流转、可统计、可告警、可留痕的核心中枢，则能尽快承载后续诸项目。
